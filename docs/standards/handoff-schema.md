# Handoff Schema — Phantom

The mandatory format for inter-agent handoffs. Every spawned agent writes a handoff before returning to the orchestrator. Every agent reading state reads the handoffs it's pointed at.

The canonical full spec lives at `~/.claude/skills/handoff/SKILL.md`. This document is a repo-local mirror of the schema agents must follow, plus Phantom-specific status values and routing fields. If the two diverge, the skill wins.

---

## Where handoffs live

```
~/.wolf/handoffs/phantom/
├── YYYY-MM-DD/
│   └── HHMMSS-{role}-{topic}.md   # one file per agent run
├── ACTIVE.md                        # rolling summary, regenerated by handoff system
└── archive/                         # entries older than 7 days
```

`{role}` is one of: `steward`, `fix`, `reviewer`, `merge`, `issue`, `prep`, `cleanup`. `{topic}` is `prN` or `issueN` or a short slug.

Filename example: `121034-steward-pr352.md` — six-digit time, role, topic.

---

## Mandatory schema

````markdown
# Handoff: {role} — {topic}

## Header

| Field | Value |
|---|---|
| Timestamp | YYYY-MM-DD HH:mm:ss TZ |
| Agent | {role} (steward / fix / reviewer / merge / issue / prep / cleanup) |
| Project | phantom |
| Issue | #{number} or "none" |
| PR | #{number} or "none" |
| Branch | {branch-name} or "none" |
| Worktree | {path} or "main" |
| Status | (see status values below) |
| Duration | Xm |

## What I Did

Bullet list. Specific. Reference files and line numbers.

- Modified `crates/phantom-foo/src/bar.rs` — [what changed]
- Ran `cargo build --workspace` — [pass/fail]
- Rebased onto `origin/main` — [N conflicts, resolved/aborted]

## Files Touched

| File | Action | Lines Changed |
|---|---|---|
| `crates/phantom-foo/src/bar.rs` | modified | 88-95, 108-119 |

For PRs, this should match `git diff origin/main...HEAD --name-only` exactly.

## State Left Behind

- **Branch status:** pushed / uncommitted / conflicts / aborted
- **Build:** clean / warnings / errors
- **Tests:** pass / fail (which) / not run
- **PR:** #{number}, mergeable: yes/no
- **Force-push:** yes (--force-with-lease) / no / N/A

## Blocked By

- [ ] PR #N must merge first (reason)
- "None" if nothing blocks

## Blocks

- Issue #N depends on this
- "None" if nothing depends

## Warnings

Cross-cutting concerns other agents must know about:

- Public API change: `Foo::bar()` return type changed
- File overlap with PR #M: same `phantom-net/src/peer.rs`
- "None" if nothing

## Next Agent Should

Specific instruction the orchestrator can act on:

- "Spawn review agent. PR is rebased, builds clean."
- "Spawn fix agent — conflicts in `agent_pane.rs` (deleted on main, replaced by module dir #345). Suggested approach: port changes into the new module structure."
- "Close PR — work already on main via #346."
````

## Status values

The `Status` field is a closed enum. The orchestrator routes on this value.

| Status | Used by | Meaning | Orchestrator action |
|---|---|---|---|
| `ready-for-review` | steward, fix | Code is rebased on main, builds clean, tests pass | Spawn reviewer |
| `needs-fix` | steward | Steward hit a conflict or issue beyond its narrow remit | Spawn fix agent with the cited issues |
| `blocked` | any | Cannot proceed without orchestrator decision | Orchestrator decides (close PR, re-spec, escalate) |
| `pass` | reviewer | Every applicable rubric check passed | Spawn merge agent |
| `pass-with-issues` | reviewer | Rubric clean except for follow-up-issue-eligible findings | Spawn issue agent(s) → merge agent |
| `fix` | reviewer | At least one blocking FAIL | Spawn fix agent, then re-review |
| `merged` | merge | PR merged successfully | Round-state update only |
| `merge-blocked` | merge | Final pre-merge check failed | Orchestrator decides (revert? fix?) |
| `done` | issue, prep, cleanup | Side task completed | No follow-up |

Any other value is a contract violation — the orchestrator treats it as `blocked`.

## Hard rules for handoff writers

1. **Never edit `ACTIVE.md`.** That file is regenerated by the handoff system from individual handoffs.
2. **Never edit a previous handoff.** Handoffs are append-only. Wrong information gets a new handoff that supersedes.
3. **The `Files Touched` table is the conflict-detection source of truth.** Be exact. Stale or missing entries cause cross-agent collisions.
4. **The `Status` field gates routing.** Do not invent statuses. If your work doesn't fit, use `blocked` and explain.
5. **The `Next Agent Should` field must be actionable in one spawn.** If it requires a discussion, it's `blocked`. Write every instruction in that field as an imperative or declarative constraint — never a question. "Next Agent Should" entries go directly into spawn prompts; see [agent-prompts.md](agent-prompts.md) for the phrasing rule and rewrite patterns.
6. **Cite file:line for every claim.** "Build is broken" is not actionable; "build fails at `crates/phantom-app/src/update.rs:412` — missing match arm for `AiAction::CheckpointReached`" is.
7. **Do not author commits as the user.** Conflict-resolution commits use `rebase: <description>`. Fix-agent commits use `fix: <description>`. Never reword existing authored commits.

## Reading handoffs

When an agent starts, the handoff hook system loads `~/.wolf/handoffs/phantom/ACTIVE.md` into context. That file is the compiled "who's working on what." Agents should not read individual handoff files unless the orchestrator points them at one.

When the orchestrator spawns a downstream agent (e.g., reviewer after steward), the spawn prompt includes the **path to the upstream handoff** so the reviewer reads only what it needs.
