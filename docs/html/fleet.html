Fleet · App-of-Apps

One Phantom process can host N autonomous apps — typically builders, one per target GitHub repo — all sharing the same event bus, brain, and substrate driver. phantom-fleet is the headless orchestrator that loads a FleetSpec from TOML and runs every entry concurrently.

Mental model

 ┌─────────────── one Phantom process ───────────────┐
 │                                                   │
 │   FleetRunner                                     │
 │      ├── LoopQueueRegistry        (Arc)           │
 │      ├── SubstrateAgentDispatcher (shared)        │
 │      ├── SubstrateDriver          (shared)        │
 │      ├── BrainHandle              (shared)        │
 │      │                                            │
 │      └── N hosted AppAdapter tasks                │
 │           ├── builder: jdmiranda/phantom          │
 │           ├── builder: jdmiranda/badass-cli       │
 │           ├── builder: your-org/your-service      │
 │           └── loop:    /path/to/.phantom/loops    │
 └───────────────────────────────────────────────────┘

Why share infrastructure

Three reasons. Brain coherence: one brain sees work across all repos and can pick the highest-utility candidate globally, not locally. Rate-limit pooling: the per-hour cap applies to the brain, not per-repo, preventing N parallel builders from all firing at once. Cost: one Claude API pool, one substrate driver, one tokio runtime — N builders for the price of one.

AppKind variants

AppKind::Buildercargo feature
A phantom-builder instance pointed at a single GitHub repo. Auto-eats the issue queue at its configured trust band. Gated behind builder-apps.
AppKind::Loop
A repo with a local .phantom/loops/*.toml directory. The fleet drives those loops directly, no GitHub dependency.
AppKind::Custom
Pluggable factory pattern. The operator provides a CustomFactory returning a BoxedLifecycle handle. Used for hosted research adapters.

FleetRunner

The orchestrator. Constructs shared state once, then spawns one tokio::task per app. Each task holds an AppAdapter trait object and a FleetContext handle. Tasks communicate with the shared brain through the same LoopQueueRegistry — so a builder for repo A can in principle enqueue work for a builder of repo B if the brain decides that's useful. 

pub struct FleetRunner { … }

impl FleetRunner {
    pub async fn run(self) -> FleetResult<()> {
        for app in &self.spec.apps {
            let ctx    = self.context_for(app);
            let handle = match &app.kind {
                AppKind::Builder(b) => spawn_builder(b, ctx).await?,
                AppKind::Loop(l)    => spawn_loop(l, ctx).await?,
                AppKind::Custom(c)  => spawn_custom(c, ctx).await?,
            };
            self.registry.register(app.id.clone(), handle);
        }
        self.registry.join_all().await
    }
}

FleetRegistry & observability

Every running app has a FleetAppHandle tracked in the FleetRegistry. Snapshots expose FleetAppStatus (Running, Paused, Errored, ExitedClean) and last-tick timestamps. A future hub surface tails the registry to render fleet-wide status. 

pub struct FleetAppSnapshot {
    pub id:             FleetAppId,
    pub kind_name:      String,
    pub status:         FleetAppStatus,
    pub started_at:     SystemTime,
    pub last_tick:      Option<SystemTime>,
    pub in_flight:      u32,
    pub enqueued_total: u64,
}

Quick start

use phantom_fleet::{FleetSpec, run_fleet};

// load spec from canonical config path
let spec = FleetSpec::load(Path::new("~/.phantom/fleet.toml"))?;
run_fleet(spec).await?;
# build with builder-apps feature
cargo build --release --features phantom-fleet/builder-apps

# run a fleet
phantom-fleet run --config ~/.phantom/fleet.toml

Compile-time gating

The builder-apps cargo feature
AppKind::Builder entries are gated behind the optional builder-apps Cargo feature on phantom-fleet. The fleet compiles cleanly without it. Trying to run a builder entry on a binary built without the feature returns a clear FleetError::Unsupported pointing at the exact rebuild command — no silent fallback.

The path forward

Today: fleet hosts builders + local loop apps. Tomorrow: the AppAdapter trait + pub/sub event bus (phantom-adapter) means anything can be hosted — a vision-capture app, a voice-driven app, a federation relay, an MCP server endpoint. Each app sees the shared event stream and can subscribe to whatever topics it cares about. The fleet is the substrate's kernel; everything above it is an app.
