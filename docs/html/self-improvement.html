Self-Improvement

Phantom wants to and can safely modify its own codebase. The brain auto-discovers work in jdmiranda/phantom through pluggable goal sources, scores each candidate with a weighted-sum utility function, applies hard exclusions and trust-band thresholds, throttles through a sliding-window rate limiter, and emits AiAction::EnqueueLoopMessage for the highest-scoring survivors. The implementer loop picks them up and ships PRs. The full pipeline is governed and auditable.

Opt-in by default
SelfImprovementConfig::enabled = false ships off. The operator must explicitly turn it on. Trust band starts at Conservative and ramps only on observed success.

2 + pluggable

0.0 – 1.0

GoalSource trait

Every "where does the brain look for things to do" surface is a GoalSource. The trait is deliberately tiny — a name and a pull. Implementations dedup on external_id so the same issue doesn't fire twice in one window. 

pub trait GoalSource {
    fn name(&self) -> &str;
    fn poll(&mut self)
        -> Result<Vec<GoalCandidate>, GoalSourceError>;
}

pub struct GoalCandidate {
    pub source:      String,
    pub external_id: String,                // "gh-issue:649"
    pub title:       String,
    pub body:        Option<String>,
    pub labels:      Vec<String>,
    pub created_at:  SystemTime,
    pub signals:     HashMap<String, f32>,  // priority, age, activity, …
}

Two concrete sources

GhIssueGoalSourcegoal_source/gh_issues.rs
Shells out to gh issue list --json …. Filters by repo + open status. Emits one candidate per matching issue with external_id = "gh-issue:{number}".
GhCiFailureGoalSourcegoal_source/gh_ci.rs
Shells out to gh run list --json …. Filters by failed status. Emits one candidate per failed CI run with synthesized title/body.

The weighted-sum scorer

score_candidate(&GoalCandidate) → (f32, HashMap<String, f32>) at self_improvement.rs:~350. Each signal contributes a weighted score; the sum is clamped to [0.0, 1.0]. The breakdown is returned alongside the score so the audit log can show why a decision was made.

SignalWeightSource
priority0.30label parser — priority:high → 1.0
recent_ci_failures0.20CI signal: how many recent CI runs touched this file
age_hours0.15time-since-opened, logistic curve
label_bonus±0.15well-defined labels add; ambiguous labels subtract
activity0.10comments/reactions in last 72h
blocked_by_count-0.10open dependencies subtract — pick unblocked work first

Critical label floor. Any candidate with a critical, regression, or blocker label has its score floored at CRITICAL_LABEL_FLOOR = 0.85 regardless of other signals. Per design doc §7.1: critical things take precedence over the scorer's opinion.

HardExclusions — categorical refusals

Some candidates are never eligible, regardless of score. A single predicate matches a Some(reason) on any such candidate. Examples: 

pub struct HardExclusions { … }

impl HardExclusions {
    pub fn matches(&self, c: &GoalCandidate) -> Option<String> {
        if c.labels.contains(&"security") { return Some("security label".into()); }
        if c.labels.contains(&"draft")    { return Some("draft".into()); }
        if c.signals.get("brain_authored") == Some(&1.0) {
            return Some("brain authored — refuse self-loops".into());
        }
        if c.signals.get("in_progress") == Some(&1.0) {
            return Some("in-progress — another agent owns this".into());
        }
        None
    }
}

TrustBudget & the four bands

Trust is an integer in [0, 20]. It starts at the operator's chosen band and ramps on enqueue success / failure. The threshold and rate limits change with the band.

SuggestionOnlybudget = 0
No auto-enqueue. The brain still scores candidates and writes audit entries, but never enqueues. Suggestions appear in the UI for human approval.
Conservativebudget 1–3
Threshold 0.85. Half the per-hour cap. Default first-week setting. Builds trust through wins.
Standardbudget 4–9
Threshold 0.75. Design-doc defaults. 4 enqueues per hour. Most ongoing operation lives here.
Aggressivebudget 10–20
Threshold 0.65. Double the per-hour cap. Reserved for proven operators. Picks up borderline work.

RateLimiter — three windows

Every enqueue is subject to three throttles simultaneously. Failure on any one short-circuits the decision to skip with reason.

per_hour
Sliding-window counter. Conservative = 2/h, Standard = 4/h, Aggressive = 8/h.
per_day
Daily ceiling. Stops runaway PR storms after off-hours drift.
cooldown
DEFAULT_COOLDOWN = 600s. Minimum time between any two enqueues.

AuditEntry — every decision logged

Every brain tick that touches a candidate writes one JSONL envelope to the configured audit log. Decisions and skips. The tail of the audit log is the brain's running mind. 

pub struct AuditEntry {
    pub timestamp:        SystemTime,
    pub external_id:      String,
    pub source:           String,
    pub score:            f32,
    pub score_breakdown:  HashMap<String, f32>,
    pub decision:         Decision,        // Enqueued · Skipped
    pub reason:           String,          // "below threshold 0.85" etc.
    pub trust_band:       TrustBand,
}
# tail the audit log
tail -f .phantom/brain/audit.jsonl | jq '.'

# count enqueues today
grep '"decision":"Enqueued"' .phantom/brain/audit.jsonl | wc -l

# last skip with reason
grep '"decision":"Skipped"' .phantom/brain/audit.jsonl | tail -1 | jq '.reason, .score_breakdown'

Why this is safe

No self-loop
HardExclusions matches brain_authored=1 — the brain refuses to score work it itself opened.
No security work
Anything labeled security is excluded categorically. Humans only.
Hard cap
Three rate-limiter windows (hour, day, cooldown) all enforce simultaneously.
Reviewer is downstream
Implementer's on_complete always enqueues to review-queue. Every brain-authored PR sees a second pair of eyes (agent or human).
Full audit
Every decision (enqueue or skip) is one JSONL line. Tail it. Diff it. Replay it.
Trust ramps slowly
Starts Conservative. One failure drops the budget. Has to earn Standard then Aggressive.

The thesis
Phantom is not a chatbot you query. It is a substrate that has goals. Right now its only goal is self-improvement — eat its own issue queue. Tomorrow its goals are yours. The same machinery — goal sources, scorer, trust bands, audit — will be pointed at your codebase, your roadmap, your bugs.
