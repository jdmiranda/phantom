Agent System

Phantom doesn't have one agent. It has a population of typed agents — 12 specialized roles, each constrained by an explicit capability manifest. Tool calls are checked at a dispatch gate the agent cannot bypass. Termination is contractual: complete_task with a schema-valid payload, or 3 strikes and the pane flatlines.

The 12 roles

Defined at phantom-agents/src/role.rs:48. Each role is a distinct specialist with a curated tool whitelist.

Conversationalturn-based
User-facing chat agent. Full LLM. Cannot directly Act — must delegate via an Actor child. This is the agent you talk to.
Watcherlong-lived
Stream observer. Reads-only. Watches for events of interest and emits structured signals. No write tools.
Captureri/o only
Pure I/O. Frames / audio / scroll-back capture. No LLM. The dumbest, most reliable worker in the mesh.
Transcribertransform
Single-purpose audio → transcript. One job. No tools beyond the STT primitive.
Reflectorperiodic
Summarizes the memory stream. Emits day-notes. Runs on a cadence, not a trigger.
Indexersubstrate
Maintains vector indexes. Tools: embedding + bundle-store writes. Never speaks to the user.
Actorshort-lived
Scoped Act executor. Spawned explicitly with consent. The only role that can mutate the user's world.
Composermulti-step
Plans and delegates. Spawned by Conversational for multi-step work. Reads, decomposes, dispatches Actors.
Fixerunblock
Short-lived scoped fixer. Spawned when an agent blocks. Reads blockage, proposes fix, exits.
Defendersecurity
Security observer auto-spawned on capability denial. Sense + Coordinate only. Never executes.
Dispatchercoordinator
Queries GitHub issues, steers agent allocation. The traffic controller for the implementer loop.
CartographerDAG nav
Conversational DAG navigator. Reads the code DAG, sends commands to the viewer surface. Issue #67.

The 5 capability axes

Orthogonal — every tool belongs to exactly one capability class. Every role has a manifest mapping the classes it may use. The capability gate enforces this at the dispatch boundary.

Sense
Read-only observation. read_file, list_dir, git_status, tail_log.
Reflect
Write substrate-internal state. memory_write, embedding_index, note_append.
Compute
LLM / embedding calls. llm_chat, embed, summarize.
Act
Mutate the user's world. write_file, run_command, gh_pr_create. The dangerous class.
Coordinate
Spawn / steer agents. spawn_subagent, enqueue_loop_message, request_review.

The dispatch gate

The single most important line of code in the agent system lives at phantom-agents/src/dispatch/mod.rs. Every tool call routes through it. There is no path around. On denial the model receives a canonical error string and self-corrects on the next turn. 

// dispatch/mod.rs — the gate
pub async fn dispatch_tool(
    ctx:  &DispatchContext,
    tool: &dyn Tool,
    args: serde_json::Value,
) -> ToolResult {
    if let Err(denied) = capability::check_capability(ctx.role, tool.class()) {
        ctx.event_log.push(Event::CapabilityDenied {
            agent_id: ctx.agent_id,
            tool:     tool.name().to_owned(),
            class:    tool.class(),
            role:     ctx.role,
        });
        return ToolResult::denied(format!(
            "capability denied: {:?} not in {:?} manifest",
            tool.class(), ctx.role,
        ));
    }
    tool.invoke(ctx, args).await
}

The event log being non-optional at this boundary (PR #655) is what closes the last audit gap. There is no construction path that produces a DispatchContext without an event log — so denials cannot be silently swallowed.

The complete_task lifecycle contract

Spawning an agent with AgentSpawnOpts::with_requires_complete_task(true) turns "the agent is done when its stream ends" into "the agent is done when it calls complete_task with a payload that validates against the ExitSchema." Strings-as-status is gone. 

let opts = AgentSpawnOpts::new(AgentRole::Actor, task)
    .with_requires_complete_task(true)
    .with_auto_approve(true)
    .with_exit_schema(schema);

// agent calls complete_task with typed payload:
// {
//   "issue_number": 642,
//   "pr_url":       "https://github.com/jdmiranda/phantom/pull/681",
//   "summary":      "Fixed the alacritty scroll-back overflow."
// }

// validation failure → validation_failure_count++
// 3 strikes → AgentPane flatline. Manual retry required.
Why this matters
Before complete_task, the substrate read agent-as-text. An agent could say "PARTIAL: I think the PR was opened, see above" and the dispatcher would shrug and pass that string downstream. After: the schema rejects "PARTIAL". The substrate refuses to advance until the agent produces a structured exit. Three failures and we don't trust the agent — we flatline and require a human to look at it.

The fast-path audit

Any time the harness skips a gate, it must be observable. PR #653 introduced Event::FastPathTaken { agent_id, kind, reason } with two variants: 

pub enum FastPathKind {
    AutoApprove,    // AwaitingApproval skipped because auto_approve=true
    SkipPlanning,   // Planning skipped because the task was simple enough
}

Every skip ends up in the event log with a human-readable reason. The audit trail says "we skipped this here for this reason." If a regression slips, you can grep the log and find the exact moment the harness gave too much rope.

Spawning an agent

# inside phantom, press ` (backtick) to open command mode, then:
agent "fix the failing tests"          # spawns Conversational
agents                                # list active agents
agent --role actor "rebuild target"   # scoped one-shot Act

The taint model

Every output an agent produces carries a taint level: Clean, AgentDerived, UserConfirmed. Downstream tools refuse to consume AgentDerived data as command input without UserConfirmed escalation. This is what prevents prompt-injection pivots — an agent reading a webpage cannot turn the webpage's contents into a run_command argument without going through the user.
