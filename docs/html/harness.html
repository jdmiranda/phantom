SECTION 07 · HARNESS CONTROL SYS://harness.0

Harness Control

Agent constraints belong in the harness, not the prompt. "Prompt it better" is never the answer because the model can always be talked out of a rule. The harness can't be talked out of anything. Phantom's design pushes every constraint that can be enforced structurally down to the substrate — tool whitelists, state machines, typed messages, schema validation — and reserves the prompt for taste, not contracts.

The anti-pattern
Every "AI agent that goes off the rails" story has the same shape: the developer's only enforcement mechanism was the system prompt. "You MUST NOT delete files outside of …" is not a constraint; it's a wish. If rm -rf / is reachable through a tool the agent has, the constraint is structural, not textual. Prompt it better is not the fix.

The four properties

All four are landed on main. The five audit gaps tracked under issue #650 are closed.

Property 1 · Per-role tool whitelists LANDED
Every tool-use routes through the capability gate at phantom-agents/src/dispatch/mod.rs: 
capability::check_capability(ctx.role, tool.class())?;
Denial returns a canonical "capability denied: <Class> not in <Role> manifest" ToolResult. PR #655 made the event log non-optional at this boundary — the gate cannot be silently bypassed.
Property 2 · External state machine LANDED
phantom-brain::TaskLedger with the try_dispatch guarded mutator. The agent cannot transition its own plan-step status — only the ledger can, and only through a typed call that returns Result<&PlanStep, DispatchBlocked>. The 9-state AgentStatus FSM makes illegal transitions unrepresentable. PR #654 + PR #657 closed the relevant gaps.
Property 3 · Structured exit LANDED
The complete_task lifecycle tool with typed JSON payload validated against the per-loop ExitSchema. The legacy stringly-typed PARTIAL exit is gone. A 3-strike validation_failure_count on the pane flatlines after three consecutive schema-invalid calls. PRs #652 + #656.
Property 4 · Typed inter-agent messaging LANDED
phantom-protocol::Event bus with 22 typed variants + EventTopic routing. FastPathTaken envelope (PR #653) audits every gate skip. phantom-loop::LoopMessage with LoopQueueRegistry handles cross-loop fan-out. Strings between agents are no longer first-class.

Why a state machine and not a "plan as text"

The model has no concept of monotonicity. Ask it "did you finish step 3?" and it will tell you yes whether or not step 3 finished. The state machine doesn't ask the model. It tracks the answer itself. PlanStep.status is a typed enum the agent cannot mutate. The only transition path is through TaskLedger::try_dispatch, which checks dependencies, status, and quarantine before allowing the change.

The fast-path audit pattern

Sometimes you want to skip a gate — auto-approving low-risk Actor calls in a trusted repo, or skipping planning for one-line tasks. The harness rule is: if you skip, audit. PR #653 introduced this pattern: 

if opts.auto_approve {
    event_log.push(Event::FastPathTaken {
        agent_id,
        kind:   FastPathKind::AutoApprove,
        reason: "caller passed AgentSpawnOpts::with_auto_approve(true)".into(),
    });
    transition_to(AgentStatus::Working);
} else {
    transition_to(AgentStatus::AwaitingApproval);
}

Now you can grep the event log for FastPathTaken and see every gate skip, with the reason the harness was given. Regressions surface as visible audit entries, not invisible silent advancements.

Why this is sufficient
The four properties form a closed system. A misbehaving agent cannot: (1) call a tool outside its role's manifest, (2) advance a plan step it shouldn't, (3) exit without a schema-valid payload, or (4) communicate with another agent through anything but a typed message. Inside that envelope the agent has full creative latitude. Outside that envelope it has no latitude at all.
