SECTION 04 · TWO-PROCESS SUPERVISION SYS://supervisor.0

Supervisor

Phantom runs as two processes. A featherweight phantom-supervisor monitors a heavyweight phantom child via Unix socket heartbeat. If the GPU hangs, the renderer panics, the terminal deadlocks, or memory pressure crosses a threshold, the supervisor sends SIGTERM, waits a second for graceful exit, escalates to SIGKILL, and respawns. Erlang would approve.

The handshake

Supervisor starts first. It creates a Unix domain socket at a per-PID path, exports PHANTOM_SUPERVISOR_SOCK, and spawns phantom as a child (phantom-supervisor/src/main.rs:186). The child reads the env var, connects, and starts the heartbeat tick. 

// phantom-protocol/src/lib.rs:29-35
pub const HEARTBEAT_TIMEOUT_MS: u64 = 10_000;
pub const MAX_RESTARTS: u32    = 5;
pub const RESTART_WINDOW_SECS: u64 = 60;
pub const GRACE_PERIOD_MS: u64 = 1_000;

The protocol

phantom-protocol defines a length-prefixed JSON wire format. Three message classes:

DirectionVariantPurpose
App → SupervisorAppMessage::HeartbeatLiveness ping. Once per second.
App → SupervisorAppMessage::EventForward an Event for monitoring/telemetry.
App → SupervisorAppMessage::ShutdownClean exit. Supervisor will not respawn.
Supervisor → AppSupervisorMessage::RestartRequest graceful restart (e.g. config reload).
Supervisor → AppSupervisorMessage::QuitFinal shutdown request.

Crash forensics

On every restart the supervisor appends a structured JSON envelope to ~/.config/phantom/supervisor_events.jsonl (phantom-supervisor/src/main.rs:46-67). The pre_sigterm_state block captures everything the supervisor knows about the dying child right before it kills it. 

struct PreSigtermState {
    reason:                  String,        // "heartbeat_timeout" | "explicit_request" | …
    pid:                     u32,
    uptime_s:                u64,
    last_heartbeat_ms_ago:   u64,
    restart_count:           u32,
    supervisor_uptime_s:     u64,
    app_connected:           bool,
}

Signal handling on the app side

phantom/src/main.rs:366-461 installs async-signal-safe handlers for SIGSEGV · SIGBUS · SIGABRT · SIGTERM. They raw_append_to_log a one-line crash summary to ~/.config/phantom/signal_crash.log before re-raising the signal. The supervisor then reads this file alongside supervisor_events.jsonl to correlate cause and effect across the process boundary.

Why two processes
Wgpu can hang. Drivers can deadlock. Tokio runtimes can wedge under memory pressure. If the supervisor lived in the same process as the renderer, all of those would also wedge the supervisor. Splitting them means something always responds. Even if the entire app process is frozen, you can still close the supervisor (a single-page tokio task) and have a clean restart.

Running with the supervisor

# recommended path — supervisor wraps phantom
cargo run --bin phantom-supervisor

# or just the convenience script
./run.sh

# standalone (no supervisor monitoring; for hard-crash debugging)
cargo run --bin phantom

Inspecting state

# tail the crash log
tail -f ~/.config/phantom/supervisor_events.jsonl | jq

# last app signal
cat ~/.config/phantom/signal_crash.log
